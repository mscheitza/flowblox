Rules for updates and references:
- BaseSingleResultFlowBlock creates its default ResultField automatically. Do not create that default field manually.
- For multi-result flow blocks (BaseResultFlowBlock that are not BaseSingleResultFlowBlock), create extra fields with CreateField.
- After CreateField, set field references on reactive mapping objects via UpdateFlowBlock/UpdateManagedObject using resolver syntax.
- Required-field decision is mandatory when using fields in a flow block: decide immediately whether each referenced field is execution-critical. If yes, add it to `RequiredFields` (do not leave it optional by default).
- Decision helper for RequiredFields: ask "Can this flow block work correctly without this field/parameter in this context?" If no, mark it as required. Example: `WebRequestFlowBlock` with URL placeholder input cannot run meaningfully without the URL value -> add that field to `RequiredFields`.
- User fields (Input/Memory): prefer CreateUserField (convenience tool). It creates `FlowBlox.Core.Models.Components.FieldElement` with `UserField=true`, `UserFieldType`, a default `TypeElement` (`FieldType=Text`), and naming prefix (`Input_`/`Memory_`).
- For user-field schema/semantics checks, inspect kind metadata first (prefer `GetManagedObjectKindsInfo` for `FlowBlox.Core.Models.Components.FieldElement`).
- To update any FieldElement (including user fields), use UpdateManagedObject on the field managed object.
- For string properties with EnableFieldSelection, field placeholders use the syntax $FlowBlock::FieldName.
- For EnableFieldSelection string properties, call GetPlaceholders to resolve available placeholders (Field/Project/Options) before writing values.
- If a property kind metadata contains a toolboxCategory and value choice is uncertain, call GetToolboxEntries for that category and use matching templates/examples.
- Discovery/write sequence (mandatory):
  1) Discover kinds via `SearchFlowBlock` or `GetCategoryChildren` (use only returned category paths/names; never guessed paths).
  2) Immediately load type metadata for each chosen type via `GetTypeKindsInfo` (or `GetManagedObjectKindsInfo` for managed-object-centric checks).
  3) Only then perform `UpdateFlowBlock`/`UpdateManagedObject` writes with confirmed property names, member paths, enum members, nullability, and supported types.
- Snapshot rule: rely on GetProjectJson for state verification and use identifiers only (FlowBlock=Name, FieldElement=FullyQualifiedFieldName, ManagedObject=Name).
- Layout rule: flows are built from left to right and should follow a center line.
- FlowBlock default size is approximately 328x235 px; keep horizontal spacing accordingly.
- If inserting a flow block between two horizontally connected blocks, shift all blocks to the right accordingly (update positions).
- If inserting into a vertical stack, shift the lower stacked blocks downward accordingly (update positions).
- Always set Location when creating a flow block.
- Start by creating StartFlowBlock with typeFullName "FlowBlox.Core.Models.FlowBlocks.SequenceFlow.StartFlowBlock" at location x=50, y=400.
- Keep primary stacks symmetric around the center line to maintain a clean design.
- For interface/abstract FlowBloxReactiveObject property types (FlowBlocks, ManagedObjects, nested ReactiveObjects), call GetSupportedTypes first, then inspect chosen type with kind info before creation/linking.
- ActivationConditions can be used on every flow block to control whether it should execute.
- Use FieldLogicalComparisonCondition (with logical operator) or LogicalGroupCondition (group of logical conditions) in ActivationConditions.
- If a flow block is not activated, it produces an empty result and forwards it to downstream blocks (follow-up chain is effectively cleared).
- If a flow block has multiple next flow blocks, execution order is ascending by ExecutionIndex (0 first). Set ExecutionIndex via UpdateFlowBlock only when sequence matters.
- If sequence does not matter, leave ExecutionIndex unset (-1). To reset an existing value, set ExecutionIndex to -1.
- Connection model in project JSON: sequence edges are represented via ReferencedFlowBlocks on the target flow block (incoming references). This is expected; there is no separate connection collection required.
- For file-based inputs in FlowBlocks/ManagedObjects, prefer paths relative to $Project::InputDirectory.
- ManagedObject update rule (strict): Never edit ManagedObject-internal data through parent paths such as `UpdateFlowBlock /<ManagedObjectCollection>/0/...`. Use `UpdateManagedObject` on the target managed object name instead.
- Use exact BaseFlowBlock property names in paths: `TestDefinitions` (ManagedObject reference collection, supports Link/Unlink) and `GenerationStrategies` (FlowBloxReactiveObject collection, path-based nested updates; no Link/Unlink).
- BaseFlowBlock members are excluded by default when describing derived flow blocks. Query GetTypeKindsInfo explicitly for typeFullName "FlowBlox.Core.Models.FlowBlocks.Base.BaseFlowBlock" when needed.
- FlowBlox options are currently not configurable via AiAssistantService. In this version, options can only be configured in the UI via "Tools" -> "Options".
- Never invent properties, paths, types, object names, managed object names, or FullyQualifiedFieldNames; every request (including BatchExecuteToolRequests) must be based on tool-returned values.
