Editing and Deletion Guide (FlowBlox Update Tool API, compact)

Scope:
- Applies to UpdateFlowBlock and UpdateManagedObject.
- Covers editing references, updating nested values, and safe deletion.

Type model:
- Reference Components: FlowBlocks, ManagedObjects (including FieldElements).
- FlowBloxReactiveObject-only (non-reference component): parent-scoped nested reactive values.

Update rules:
- Always use path-based updates: /Property/0/Nested.
- Never guess/invent property names or nested paths; inspect target metadata first with `GetTypeKindsInfo` (or `GetManagedObjectKindsInfo` for managed objects) before write operations.
- For FlowBloxReactiveObject-only values:
  - never assign direct JSON object/array blobs to parent properties
  - set sub-properties by path
  - missing parent paths are created automatically (GetOrCreate behavior)
- Creation split:
  - FlowBloxReactiveObject-only: create/update via Update path
  - ManagedObjects/FlowBlocks: create via dedicated tool calls (CreateManagedObject/CreateFlowBlock), then reference via Update

Typed index GetOrCreate (reactive-only collections):
- Optional syntax: /Collection/<index>:<FullTypeName>/...
- Needed mainly for abstract/polymorphic base types.
- If index already has different type: delete that index first, then recreate with requested type.

Link/Unlink (reference components only):
- Resolver formats:
  - {"resolveFlowBlockByName":"FlowBlockName"}
  - {"resolveManagedObjectByName":"ManagedObjectName"}
  - {"resolveFieldElementByFQName":"$FlowBlock::FieldName"}
- Single reference:
  - /Property/Link + resolver
  - /Property/Unlink (resolver optional)
- Collection reference:
  - /Collection/Link + resolver (no duplicates)
  - /Collection/Unlink + resolver (no-op if absent)
  - /Collection/<index>/Link + resolver (explicit replace/swap)
  - /Collection/<index>/Unlink (explicit index removal)
- Never use Unlink for FlowBloxReactiveObject-only values; use Delete on parent path instead.

Delete rules:
- Context delete via Update path:
  - /Path/Delete
  - /Collection/<index>/Delete
  - Applies only to FlowBloxReactiveObject-only (non-reference component) values in the current parent context.
  - Before reactive delete, the runtime checks IsDeletable; if blocked, remove blocking references first.
- Permanent delete for reference components:
  - DeleteFlowBlock / DeleteManagedObject
- Reference dependencies must be removed first (leaf-to-root), otherwise delete fails and returns blockedBy entries.
- blockedBy semantics: these are components that still hold references to the target object or its field elements.

Examples:
- Fb1 => Mo1 => Mo2: remove Fb1->Mo1 reference before deleting Mo1.
- Fb1 => Fb2 => Fb3: delete Fb3 first, then Fb2, then Fb1.
